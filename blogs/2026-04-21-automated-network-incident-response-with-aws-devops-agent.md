<p>Your on-call engineer gets paged at 2 AM. A payment service in Workload Account cannot reach a shared database in Shared Services Account. The <a href="https://aws.amazon.com/cloudwatch/" rel="noopener noreferrer" target="_blank">Amazon CloudWatch</a> alarm fired eight minutes ago. The engineer starts by checking route tables across two accounts, <a href="https://aws.amazon.com/vpc/" rel="noopener noreferrer" target="_blank">Amazon Virtual Private Cloud (Amazon VPC)</a> attachment states, <a href="https://docs.aws.amazon.com/vpc/latest/userguide/vpc-security-groups.html" rel="noopener noreferrer" target="_blank">security group</a> rules on both sides, <a href="https://docs.aws.amazon.com/vpc/latest/userguide/vpc-network-acls.html" rel="noopener noreferrer" target="_blank">network ACLs</a>, and DNS resolution logs. An hour later, the root cause turns out to be a <a href="https://aws.amazon.com/transit-gateway/" rel="noopener noreferrer" target="_blank">AWS Transit Gateway</a> route table association that was swapped to the wrong route table during a network migration earlier that evening, quietly dropping all traffic between the two VPCs while every other spoke kept working fine.</p> 
<p>AWS DevOps Agent solves this problem by automating the entire investigation process. When a CloudWatch alarm fires, DevOps Agent receives the alert through a webhook, then correlates metrics, logs, network flow data, and API change history. DevOps Agent then provides a root cause analysis with a ready-to-run fix, reducing that hour of manual correlation to minutes of automated analysis.</p> 
<p>This post demonstrates how to integrate CloudWatch monitoring with DevOps Agent to enable automated incident response for network failures. We walk through four common scenarios: security group misconfigurations, NAT Gateway routing issues, VPC endpoint policy restrictions, and Interface Endpoint subnet problems. We provide a CloudFormation template you can deploy in your own account to follow along, with each scenario showing how DevOps Agent automatically investigates and generates remediation plans with human oversight. We then show a complex multi-account scenario where a Transit Gateway route table misconfiguration blocks traffic between VPCs across accounts, demonstrating how DevOps Agent scales to enterprise environments. The full stack of the simulated workload application is available on <a href="https://github.com/aws-samples/sample-automated-aws-devops-agent-network-incident-response" rel="noopener noreferrer" target="_blank">aws-samples GitHub repo</a>.</p> 
<h2>The simulated workload</h2> 
<p>To demonstrate DevOps Agent’s automated incident response capabilities, we use a simulated workload: a Node.js service running on Amazon Elastic Compute Cloud (Amazon EC2) behind an Application Load Balancer (ALB). This application performs health checks on a loop and displays results on a status dashboard. For the sake of simplicity and illustration, the application checks connectivity to four types of resources:</p> 
<ul> 
 <li>Database connectivity to an Amazon RDS instance (every 5 seconds).</li> 
 <li>Outbound internet reachability through a NAT Gateway (every 15 seconds).</li> 
 <li>Amazon Simple Storage Service (Amazon S3) bucket access through a VPC Gateway Endpoint (every 10 seconds).</li> 
 <li>Amazon Bedrock access through an interface VPC Endpoint (every 60 seconds).</li> 
</ul> 
<p>When a check fails, the application publishes a custom Amazon CloudWatch metric that triggers an alarm. The whole stack deploys from a single AWS CloudFormation template. Figure 1 shows the architecture diagram of the simulated workload.</p>

<h2>How the alarm pipeline works, from failure to investigation</h2> 
<p>Every failure follows the same path to DevOps Agent. The application publishes a custom metric to Amazon CloudWatch. A CloudWatch alarm evaluates that metric and transitions to ALARM. The alarm notifies an <a href="https://aws.amazon.com/sns/" rel="noopener noreferrer" target="_blank">Amazon SNS</a> topic. Amazon SNS invokes an <a href="https://aws.amazon.com/lambda/" rel="noopener noreferrer" target="_blank">AWS Lambda</a> function that reads webhook credentials from <a href="https://aws.amazon.com/secrets-manager/" rel="noopener noreferrer" target="_blank">AWS Secrets Manager</a>, builds an incident payload, signs it with HMAC SHA-256, and POSTs it to the DevOps Agent webhook. DevOps Agent validates the signature and starts an investigation. This pipeline is shown in Figure 4.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-4.png" rel="noopener" target="_blank"><img alt="Alarm pipeline" class="aligncenter wp-image-32849 size-full" height="270" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-4.png" width="1149" /></a><em>Figure 4: Alarm pipeline</em></p> 
<p>Figure 5 shows the four CloudWatch alarms configured to monitor the simulated workload application.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-5.png" rel="noopener" target="_blank"><img alt="CloudWatch Alarms" class="aligncenter wp-image-32850 size-full" height="708" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-5.png" width="1844" /></a><em>Figure 5: CloudWatch Alarms</em></p> 
<p>Amazon SNS sits between Amazon CloudWatch and AWS Lambda for three reasons: it retries delivery if Lambda is throttled (with dead letter queue support), it fans out to additional subscribers (email, chat tools, incident management systems) without touching the alarm, and it supports cross account publishing so alarms from multiple accounts can feed a single operations pipeline.</p> 
<h2>Scenario 1: Security group rule deleted, database goes down</h2> 
<p>The database security group (<code>simulated-workload-DB-SG</code>) allows inbound MySQL traffic (port 3306) from the application security group (<code>devops-agent-sample-app-App-SG</code>). The following Figure 6 shows this the inbound rule being deleted.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-6.gif" rel="noopener" target="_blank"><img alt="Security group inbound rule being deleted" class="aligncenter wp-image-32851 size-full" height="682" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-6.gif" width="1504" /></a><em>Figure 6: Security group inbound rule being deleted</em></p> 
<p>When someone deletes the inbound rule, the app loses database connectivity within seconds. Security groups silently drop packets that do not match any rule, so the application’s connection attempt gets no response and times out (you will see <code>ETIMEDOUT</code> on the status dashboard). The other three checks (external connectivity, S3, and Bedrock) stay green because none of them touch port 3306.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-7.png" rel="noopener" target="_blank"><img alt="Simulated workload application status page" class="aligncenter wp-image-32852 size-full" height="1584" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-7.png" width="1856" /></a><em>Figure 7: Simulated workload application status page</em></p> 
<p>DevOps Agent receives the <code>DatabaseConnectionFailures</code> alarm and investigates:</p> 
<ol> 
 <li><strong>Checks RDS instance health</strong> – Confirms the database is available but reporting zero connections.</li> 
 <li><strong>Searches CloudTrail logs</strong> – Finds the <code>RevokeSecurityGroupIngress</code> API call, including who ran it and when (just 15 seconds before the alarm triggered).</li> 
 <li><strong>Maps resource relationships</strong> – Uses its topology to identify the EC2-to-RDS connection dependency.</li> 
 <li><strong>Identifies root cause</strong> – The inbound rule on the database security group that allowed traffic from the application security group on port 3306 was removed, immediately blocking all MySQL connections while the database itself remained healthy.</li> 
</ol>

<h2>Scenario 2: NAT Gateway route deleted, outbound internet lost</h2> 
<p>The private route table (<code>simulated-workload-Private-RT</code>) has a default route that sends all non-VPC traffic to the NAT Gateway. The following Figure 11 shows the route being deleted.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-11.gif" rel="noopener" target="_blank"><img alt="Default route deleted from the private route table" class="aligncenter wp-image-32857 size-full" height="680" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-11.gif" width="1506" /></a><em>Figure 11: Default route deleted from the private route table</em></p> 
<p>When someone deletes this route, the app can no longer reach the public internet. But the database still works (VPC local route), Amazon S3 still works (Gateway Endpoint has its own route table entry), and Amazon Bedrock still works (Interface Endpoint places ENIs directly in the subnet). Only the external connectivity check turns red.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-12.png" rel="noopener" target="_blank"><img alt="Simulated workload application status page" class="aligncenter wp-image-32858 size-full" height="1636" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-12.png" width="1854" /></a><em>Figure 12: Simulated workload application status page</em></p> 
<p>DevOps Agent receives the <code>ExternalConnectivityFailures</code> alarm and investigates:</p> 
<ol> 
 <li><strong>Examines the private route table</strong> – Identifies that the default route (0.0.0.0/0) pointing to the NAT Gateway is missing.</li> 
 <li><strong>Confirms NAT Gateway health</strong> – Verifies the NAT Gateway itself is healthy with no capacity issues or errors.</li> 
 <li><strong>Searches CloudTrail logs</strong> – Finds the <code>DeleteRoute</code> API call, including who ran it and when.</li> 
 <li><strong>Maps resource relationships</strong> – Uses its topology to identify the EC2-to-NAT Gateway dependency.</li> 
 <li><strong>Identifies root cause</strong> – The default route was removed from the private route table, leaving the EC2 instance with no routing path for internet-bound traffic.</li> 
</ol>

<h2>Scenario 3: VPC endpoint policy restricts S3 bucket access</h2> 
<p>The S3 VPC Gateway Endpoint (<code>simulated-workload-S3-Endpoint</code>) has a policy that allows access to all five application buckets. The following Figure 16 shows the modification of the VPC endpoint policy.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-16.gif" rel="noopener" target="_blank"><img alt="Modifying of the VPC endpoint policy" class="aligncenter wp-image-32862 size-full" height="680" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-16.gif" width="1504" /></a><em>Figure 16: Modifying of the VPC endpoint policy</em></p> 
<p>When someone edits this policy to allow only three of the five buckets, the other two start returning <code>AccessDenied</code>. This is the scenario that confuses people most, because the <a href="https://aws.amazon.com/iam/" rel="noopener noreferrer" target="_blank">AWS Identity and Access Management (IAM)</a> policy still grants access to all five. For Amazon S3 traffic through a Gateway Endpoint, three layers of policy have to agree: IAM, the bucket policy, and the endpoint policy. If any one of those layers denies the request, you get an Access Denied (HTTP 403 Forbidden) error.</p> 
<p>But everything else still works:</p> 
<ul> 
 <li>The database works (VPC local route).</li> 
 <li>Amazon Bedrock works (Interface Endpoint places ENIs directly in the subnet).</li> 
 <li>Outbound internet connectivity works (NAT Gateway route is intact).</li> 
</ul> 
<p>Only the S3 bucket access checks turn red.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-17.png" rel="noopener" target="_blank"><img alt="Simulated workload application status page" class="aligncenter wp-image-32863 size-full" height="1600" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-17.png" width="1868" /></a><em>Figure 17: Simulated workload application status page</em></p> 
<p>DevOps Agent receives the <code>S3AccessFailures</code> alarm and investigates:</p> 
<ol> 
 <li><strong>Compares policy layers</strong> – Examines the endpoint policy against the IAM permissions and the buckets the app tries to reach.</li> 
 <li><strong>Identifies the blocking layer</strong> – Determines the endpoint policy is restricting access.</li> 
 <li><strong>Searches CloudTrail logs</strong> – Finds the <code>ModifyVpcEndpoint</code> API call, including who modified the policy and when.</li> 
 <li><strong>Maps resource relationships</strong> – Uses its topology to identify the EC2-to-S3 Gateway Endpoint dependency.</li> 
 <li><strong>Identifies root cause</strong> – The VPC endpoint policy was modified to allow access to only three of the five application buckets, blocking access to the remaining two buckets.</li> 
</ol>

Figure 20 shows the mitigation plan detailing the steps (AWS CLI Command) needed to restore access to all five S3 buckets.</p>

<h2>Scenario 4: Bedrock Interface Endpoint subnets removed, AI feature breaks</h2> 
<p>The Bedrock Runtime Interface Endpoint (<code>simulated-workload-Bedrock-Endpoint</code>) creates ENIs in two private subnets. Private DNS resolves <code>bedrock-runtime.us-east-1.amazonaws.com</code> to those ENI IP addresses. The following Figure 21 shows the removal of subnet association.</p> 
<p><em><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-21.gif" rel="noopener" target="_blank"><img alt="Modify subnets associated with interface VPC endpoint" class="aligncenter wp-image-32867 size-full" height="680" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-21.gif" width="1506" /></a>Figure 21: Modify subnets associated with interface VPC endpoint </em></p> 
<p>When someone removes both subnet associations, the ENIs are deleted. The endpoint still exists and Private DNS still resolves the hostname to the endpoint, but there is nothing to handle the traffic. This is the scenario that is particularly hard to spot by hand, because the console still shows the endpoint as “Available” even though requests time out with no response. It is worth noting the difference between the two endpoint types: VPC Gateway Endpoints (Amazon S3, Amazon DynamoDB) work through route table entries and endpoint policies, so a misconfiguration there returns an Access Denied (HTTP 403 Forbidden) error. Interface VPC Endpoints (Amazon Bedrock and many other AWS services) work through ENIs in your subnets with their own security groups, so when those ENIs are gone, you get connection timeouts instead. But the database still works (VPC local route), Amazon S3 still works (VPC Gateway Endpoint has its own route table entry), and outbound internet connectivity still works (NAT Gateway route is intact). Only the Bedrock AI check turns red.</p> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-22.png" rel="noopener" target="_blank"><img alt="Simulated workload application status page" class="aligncenter wp-image-32868 size-full" height="1606" src="https://d2908q01vomqb2.cloudfront.net/5b384ce32d8cdef02bc3a139d4cac0a22bb029e8/2026/04/17/NetCDNBlog-1723-image-22.png" width="1872" /></a><em>Figure 22: Simulated workload application status page</em></p> 
<p>DevOps Agent receives the <code>BedrockAccessFailures</code> alarm and investigates:</p> 
<ol> 
 <li><strong>Checks endpoint configuration</strong> – Finds zero subnet associations on the Interface VPC Endpoint.</li> 
 <li><strong>Searches CloudTrail logs</strong> – Finds the <code>ModifyVpcEndpoint</code> API call, including who removed the subnet associations and when.</li> 
 <li><strong>Maps resource relationships</strong> – Uses its topology to identify the EC2-to-Bedrock Interface VPC Endpoint dependency.</li> 
 <li><strong>Identifies root cause</strong> – Both subnet associations were removed from the Bedrock Runtime Interface VPC Endpoint, deleting the ENIs that handle traffic while the endpoint itself remained in an “Available” state.</li> 
</ol>

<h2>Further considerations</h2> 
<p>In production, a single infrastructure change can fire multiple alarms at once. For example, deleting a NAT Gateway route can trigger external connectivity, Amazon S3, and Amazon Bedrock alarms simultaneously if those services also depend on outbound access. Adding correlation logic in the AWS Lambda function (for example, buffering alarms in <a href="https://aws.amazon.com/dynamodb/" rel="noopener noreferrer" target="_blank">Amazon DynamoDB</a> with a 60 second window and grouping by application) prevents duplicate investigations from piling up.</p> 
<p>The Amazon SNS topic in this design has one subscriber (the webhook Lambda function), but you can add email, <a href="https://aws.amazon.com/sqs/" rel="noopener noreferrer" target="_blank">Amazon SQS</a>, or HTTP subscribers without changing any alarm definition. If you already have CloudWatch alarms and an SNS topic powering your existing monitoring and alerting, you can add the webhook Lambda as an additional subscriber to that topic, giving DevOps Agent visibility without disrupting your current tooling. For multi-account environments, Amazon CloudWatch alarms in other accounts can publish directly to a central Amazon SNS topic, giving you a single pipeline into DevOps Agent.</p> 
<p>It is worth reiterating that DevOps Agent produces a mitigation plan but does not make changes to your AWS environment on its own. Any remediation steps still require human action.</p> 
<p>The third-party tools and data sources you connect during setup also apply during investigations:</p> 
<ul> 
 <li>If you connect a <a href="https://docs.aws.amazon.com/devopsagent/latest/userguide/configuring-capabilities-for-aws-devops-agent-connecting-telemetry-sources-index.html" rel="noopener noreferrer" target="_blank">telemetry source</a>, DevOps Agent can pull metrics and traces as part of its root cause analysis.</li> 
 <li>If you connect a <a href="https://docs.aws.amazon.com/devopsagent/latest/userguide/configuring-capabilities-for-aws-devops-agent-connecting-to-cicd-pipelines-index.html" rel="noopener noreferrer" target="_blank">CI/CD pipeline</a>, it can correlate recent deployments with the failure timeline.</li> 
 <li>If you configure <a href="https://docs.aws.amazon.com/devopsagent/latest/userguide/configuring-capabilities-for-aws-devops-agent-connecting-mcp-servers.html" rel="noopener noreferrer" target="_blank">MCP servers</a>, you can expose internal APIs, runbook databases, or any custom data source over the Model Context Protocol. DevOps Agent calls those tools during an investigation when the data is relevant.</li> 
</ul> 
<h2>Cleanup</h2> 
<p>To remove all resources, follow <a href="https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/cfn-console-delete-stack.html" rel="noopener noreferrer" target="_blank">deleting a stack</a>. The stack deletion removes all resources including the VPC, subnets, route tables, security groups, endpoints, EC2 instance, RDS, ALB, SNS topic, Lambda function, and IAM roles.</p> 
<h2>Conclusion</h2> 
<p>This post walked you through four networking scenarios where AWS DevOps Agent automatically investigates failures and produces remediation plans, giving your team clear, actionable root cause analysis and ready-to-run fixes in minutes instead of hours of manual log correlation. Each scenario demonstrates a different troubleshooting pattern so you can apply the same approach to your own environment.</p> 
<p>You saw how DevOps Agent correlates VPC Flow Logs with CloudTrail API changes to find deleted security group rules, maps route table configurations to identify missing NAT Gateway routes, compares multi-layer policies to isolate S3 Gateway Endpoint restrictions, and distinguishes between endpoint availability states to catch Interface Endpoint subnet removals. Beyond these foundational scenarios, DevOps Agent scales to complex environments with multi-account access, third-party observability integrations, and custom MCP servers.</p> 
<p>Start automating your incident response today by following the <a href="https://docs.aws.amazon.com/devopsagent/latest/userguide/getting-started-with-aws-devops-agent.html" rel="noopener noreferrer" target="_blank">Getting Started with AWS DevOps Agent</a> guide.</p>
